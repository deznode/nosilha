## Findings

| # | Severity | Description | Evidence / Files | Suggested Fix |
|---|----------|-------------|------------------|---------------|
| 1 | High | `ReactionService.getReactionCounts` caches per-content DTOs that include a `userReaction` field, but the cache key only uses `contentId`. The first caller (possibly authenticated) pollutes the cache, causing every subsequent visitor to see that user’s reaction highlighted and leaking private data. | `backend/src/main/kotlin/com/nosilha/core/contentactions/ReactionService.kt:210-244` | ✅ Fixed: cache only aggregated counts via `CacheManager` while looking up `userReaction` per request so cached payloads remain user-agnostic. |
| 2 | Medium | POST `/api/v1/reactions` always returns `201 Created` whenever the resulting reaction count stays positive, even when the user actually removed their reaction (toggle-off case). The controller currently infers operation type from `count > 0`, which is unrelated to whether we deleted or inserted a row. | `backend/src/main/kotlin/com/nosilha/core/contentactions/ReactionController.kt:95-129` | ✅ Fixed: `ReactionService.submitReaction` now returns an operation enum (`CREATED/UPDATED/REMOVED`) and the controller maps it to `201` vs `200`. |
| 3 | High | DELETE `/api/v1/reactions/content/{id}` rethrows `NoSuchElementException`, which the global handler does not map to 404, so the API returns a 500 whenever the user tries to delete a non-existent reaction. | `backend/src/main/kotlin/com/nosilha/core/contentactions/ReactionController.kt:155-172` | ✅ Fixed: service now throws `ResourceNotFoundException`, letting the global handler produce the standard 404 error payload. |
| 4 | Medium | The in-memory rate limiter stores mutation timestamps in a plain `MutableList` that multiple threads mutate concurrently (`removeIf` + `add`). Simultaneous submissions for the same user can throw `ConcurrentModificationException` or allow races that bypass the limit. | `backend/src/main/kotlin/com/nosilha/core/contentactions/ReactionService.kt:42-45`, `264-281` | ✅ Fixed: swapped to `ConcurrentLinkedDeque` for per-user windows so mutations are thread-safe until Redis backing is implemented. |
| 5 | Low | `SuggestionControllerTest` relies on Kotlin `assert` statements to verify database writes. Kotlin assertions are disabled in production/CI unless the JVM is launched with `-ea`, so those checks silently never run and the test can pass while functionality regresses. | `backend/src/test/kotlin/com/nosilha/core/contentactions/SuggestionControllerTest.kt:73-77`, `105-107` | ✅ Fixed: migrated to AssertJ `assertThat` assertions so checks run regardless of JVM flags. |
| 6 | Medium | Controller responses aren’t consistently wrapped with the shared `ApiResponse`/`PagedApiResponse` envelopes (and errors with `ErrorResponse`/`ValidationErrorResponse`). The API standards/reference never call out the requirement defined in `com/nosilha/core/shared/api/ApiResponse.kt`, so new endpoints frequently leak raw DTOs or ad-hoc maps, making clients handle multiple shapes. | `docs/API_CODING_STANDARDS.md`, `docs/API_REFERENCE.md`, `backend/src/main/kotlin/com/nosilha/core/shared/api/ApiResponse.kt` | Update the API coding standards and reference docs to mandate the shared response envelopes (success + error) and audit controllers to ensure they return the wrapper types everywhere. |
