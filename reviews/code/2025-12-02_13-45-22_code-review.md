## Critical Issues (Must Address Before Merge)

### 1. [Critical/Blocker] Analytics Environment Variable Exposure Risk

**File:** `frontend/src/app/layout.tsx` (Lines 94-95)

**Issue:**
```typescript
const GA_ID = process.env.NEXT_PUBLIC_GA_ID!;
const CLARITY_PROJECT_ID = process.env.NEXT_PUBLIC_CLARITY_PROJECT_ID || "";
```

**Problems:**
1. **Non-null assertion (`!`)** on GA_ID without runtime validation - app will crash if env var is missing
2. **Missing validation** - these values are used directly in production without checking if they're valid IDs
3. **Silent failure risk** - Clarity uses empty string fallback, which may cause unexpected behavior

**Why This is Critical:**
- **Principle Violated:** Fail-Fast (KISS) - Should validate environment at startup
- **Production Risk:** Missing or malformed analytics IDs could cause runtime errors or silent tracking failures
- **Security:** Invalid IDs might expose internal configuration errors to users

**Recommended Fix:**
```typescript
// At top of file or in a separate config validation utility
const GA_ID = process.env.NEXT_PUBLIC_GA_ID;
const CLARITY_PROJECT_ID = process.env.NEXT_PUBLIC_CLARITY_PROJECT_ID;

// Validate in development
if (process.env.NODE_ENV === 'development') {
  if (!GA_ID) {
    console.warn('[Analytics] GA_ID not configured - analytics disabled');
  }
  if (!CLARITY_PROJECT_ID) {
    console.warn('[Analytics] CLARITY_PROJECT_ID not configured - clarity disabled');
  }
}

// Then use conditional rendering:
{GA_ID && <GoogleAnalytics gaId={GA_ID} />}
{CLARITY_PROJECT_ID && <Script id="microsoft-clarity" ... />}
```

**Alternative Approach:**
Create a config validation utility at build time:
```typescript
// lib/analytics-config.ts
export const analyticsConfig = {
  gaId: process.env.NEXT_PUBLIC_GA_ID || null,
  clarityId: process.env.NEXT_PUBLIC_CLARITY_PROJECT_ID || null,
  isEnabled: !!(process.env.NEXT_PUBLIC_GA_ID || process.env.NEXT_PUBLIC_CLARITY_PROJECT_ID)
} as const;
```

### 2. [Critical/Blocker] Signup Form Lacks Validation & Proper Error Handling

**File:** `frontend/src/components/auth/signup-form.tsx`

**Issue:**
The signup form uses basic HTML validation instead of the robust Zod schema validation used in the login form.

**Problems:**
1. **Inconsistent validation** - Login uses `react-hook-form` + Zod, signup uses native HTML validation
2. **Weak password validation** - Only `minLength={6}` is insufficient for security
3. **No email format validation** - Relies on browser-native validation (inconsistent across browsers)
4. **No animation/UX parity** - Login has sophisticated Framer Motion animations, signup is static
5. **Missing loading state** - Uses boolean `isSubmitting` instead of proper loading spinner like login

**Why This is Critical:**
- **Principle Violated:** DRY, Consistency, Security Best Practices
- **Security Risk:** Weak password requirements (only 6 characters) invite credential stuffing attacks
- **UX Inconsistency:** Users experience different validation behaviors between login/signup
- **Maintainability:** Two different validation approaches increase cognitive load

**Recommended Fix:**
Refactor to match login-form.tsx pattern:

```typescript
// 1. Create signup schema (similar to loginSchema)
// schemas/authSchema.ts
export const signupSchema = z.object({
  email: z.string().trim().email("Please enter a valid email address"),
  password: z
    .string()
    .min(8, "Password must be at least 8 characters")
    .regex(/[A-Z]/, "Password must contain at least one uppercase letter")
    .regex(/[a-z]/, "Password must contain at least one lowercase letter")
    .regex(/[0-9]/, "Password must contain at least one number"),
  confirmPassword: z.string(),
}).refine((data) => data.password === data.confirmPassword, {
  message: "Passwords don't match",
  path: ["confirmPassword"],
});

// 2. Update SignupForm to use react-hook-form + Zod
// Add Framer Motion animations for consistency
// Add LoadingSpinner for better UX
```

**Additional Security Consideration:**
Consider implementing rate limiting on the signup endpoint (backend) to prevent enumeration attacks.

### 3. [Improvement] Newsletter Server Action Has Retry Logic Issues

**File:** `frontend/src/app/actions/newsletter.ts` (Lines 98-212)

**Issue:**
The retry logic lacks proper backoff strategy.

**Problems:**
1. **No exponential backoff** - Retries happen immediately, which can overwhelm the Resend API
2. **Timeout handling** - 5-second timeout might be too aggressive for slow networks

**Recommended Improvement:**
```typescript
const TIMEOUT_MS = 10000; // Increase to 10s for better mobile support
const RETRY_DELAYS = [1000, 2000]; // Exponential backoff: 1s, 2s

for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
  try {
    // ... existing code
  } catch (error) {
    if (attempt < MAX_RETRIES && shouldRetry) {
      const delay = RETRY_DELAYS[attempt] || RETRY_DELAYS[RETRY_DELAYS.length - 1];
      await new Promise(resolve => setTimeout(resolve, delay));
      continue;
    }
  }
}
```

### 4. [Improvement] Animation System Lacks Reduced Motion Support in Many Components

**Files:**
- `frontend/src/components/auth/login-form.tsx`
- `frontend/src/features/map/components/InteractiveMap.tsx`
- Multiple landing page components

**Issue:**
The centralized animation system has `useMotionConfig()` hook for reduced motion detection, but many components don't use it.

**Example from login-form.tsx:**
```typescript
<motion.div
  initial={{ opacity: 0, y: 20 }}
  animate={{ opacity: 1, y: 0 }}
  transition={{ duration: 0.4, ease: "easeOut" }}
>
```

**Why This Matters:**
- **Accessibility (WCAG 2.1):** Users with vestibular disorders need reduced motion
- **UX:** Animations can cause nausea, dizziness, or disorientation for some users

**Recommended Pattern:**
```typescript
import { useMotionConfig } from "@/lib/animation/config";

export function LoginForm() {
  const { reducedMotion } = useMotionConfig();

  return (
    <motion.div
      initial={reducedMotion ? {} : { opacity: 0, y: 20 }}
      animate={reducedMotion ? {} : { opacity: 1, y: 0 }}
      transition={reducedMotion ? {} : { duration: 0.4, ease: "easeOut" }}
    >
```

### 6. [Improvement] Missing Error Boundaries for React 19.2

**Issue:**
After upgrading to React 19.2, the app should have error boundaries to handle runtime errors gracefully.

**Current State:**
No error boundaries detected in:
- `frontend/src/app/layout.tsx`
- Feature modules (map, events)
- Content pages

**Recommended Implementation:**
```typescript
// components/error-boundary.tsx
'use client';

import { Component, ReactNode } from 'react';

export class ErrorBoundary extends Component<
  { children: ReactNode; fallback?: ReactNode },
  { hasError: boolean; error?: Error }
> {
  constructor(props) {
    super(props);
    this.state = { hasError: false };
  }

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo) {
    console.error('[ErrorBoundary]', error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || (
        <div className="error-container">
          <h2>Something went wrong</h2>
          <button onClick={() => window.location.reload()}>Refresh Page</button>
        </div>
      );
    }

    return this.props.children;
  }
}
```

### 7. [Improvement] Husky Pre-Commit Hook May Block Commits on Large Diffs

**File:** `frontend/package.json` (Lines 181-192)

**Issue:**
The lint-staged configuration runs TypeScript validation and Prettier on every commit, which can be slow for large diffs.

**Recommended Optimization:**
```javascript
// .husky/pre-commit
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"

# Skip pre-commit if >100 files changed
CHANGED_FILES=$(git diff --cached --name-only | wc -l)
if [ "$CHANGED_FILES" -gt 100 ]; then
  echo "⚠️  Skipping pre-commit hooks (>100 files changed)"
  echo "Remember to run 'npm run lint:all' manually!"
  exit 0
fi

npx lint-staged
```
