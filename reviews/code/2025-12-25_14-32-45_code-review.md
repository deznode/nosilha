### Suggested Improvements

#### [Improvement] `backend/src/main/kotlin/com/nosilha/core/contentactions/StoryService.kt:63-66`

**Silent honeypot handling breaks audit trail**

The honeypot spam protection returns a null ID, which silently accepts the request without creating a record. This prevents tracking spam patterns over time.

```kotlin
// Current implementation
if (!request.honeypot.isNullOrBlank()) {
    logger.warn("Honeypot triggered for story submission from IP: {}", ipAddress)
    return StorySubmittedResponse(id = null)
}
```

**Rationale**: According to the **Pragmatic Quality** framework, observability and debugging capabilities are high-priority concerns. The current approach sacrifices audit trail for simplicity. While the security mechanism is sound, the lack of persistent spam tracking limits your ability to:
- Identify spam patterns and IP ranges
- Tune rate limiting thresholds based on actual abuse data
- Generate security reports for compliance/analysis

**Recommendation**: Consider persisting spam attempts to a dedicated `spam_submissions` table or logging them to a structured audit log. This maintains the UX benefit (spammer sees "success") while preserving forensic capabilities.

#### [Improvement] `backend/src/main/kotlin/com/nosilha/core/contentactions/SuggestionService.kt:63-65`

**Inconsistent spam handling between services**

`SuggestionService` throws `HoneypotSpamDetectedException` for honeypot triggers, while `StoryService` returns silently. This architectural inconsistency violates the **Single Responsibility Principle** and creates maintenance burden.

```kotlin
// SuggestionService throws exception
if (!dto.honeypot.isNullOrBlank()) {
    logger.warn("Honeypot field detected - potential spam from IP: $ipAddress")
    throw HoneypotSpamDetectedException("Spam submission detected")
}

// StoryService returns null ID
if (!request.honeypot.isNullOrBlank()) {
    logger.warn("Honeypot triggered for story submission from IP: {}", ipAddress)
    return StorySubmittedResponse(id = null)
}
```

**Rationale**: **DRY (Don't Repeat Yourself)** and **consistency** are fundamental engineering principles. Different spam handling strategies across similar services create:
- Cognitive overhead for future maintainers
- Potential security gaps if only one approach is updated during a security review
- Inconsistent API behavior (suggestions return 500, stories return 201 with null ID)

**Recommendation**:
1. Extract spam protection to a shared `@Component` like `SpamProtectionService`
2. Standardize on a single approach (silent acceptance is generally better UX for honeypots)
3. Centralize spam logging for consistent audit trails

#### [Improvement] `backend/src/main/kotlin/com/nosilha/core/contentactions/api/AdminStoryController.kt:158`

**Potential null safety issue with authentication.name**

The authentication subject extraction pattern `authentication.name` could theoretically be null, though Spring Security typically guarantees non-null for authenticated requests. The recent JWT filter change (lines 68-69 in `JwtAuthenticationFilter.kt`) added null safety, but controllers don't validate.

```kotlin
val adminId = authentication.name  // Could be null if JWT filter changes
```

**Rationale**: **Defense in depth** is a security principle. While the current implementation is safe due to Spring Security's guarantees, Kotlin's null safety should be leveraged to make contracts explicit. The recent change to `JwtAuthenticationFilter` (adding `?: return` for null subject) shows awareness of this concern.

**Recommendation**: Use Kotlin's null-safe operators consistently:

```kotlin
val adminId = authentication.name ?: throw IllegalStateException("Authentication subject is missing")
```

This makes the contract explicit, improves debugging (clear error vs NPE), and aligns with Kotlin idioms.

fun countByIpAddressAndCreatedAtAfter(ipAddress: String, after: Instant): Long

#### [Improvement] `backend/src/main/kotlin/com/nosilha/core/auth/security/SecurityConfig.kt:71-75`

**Admin endpoint security could use method-level annotations for defense in depth**

Currently admin endpoints rely solely on path-based security (`/api/v1/admin/**` requires ADMIN role). While correct, adding method-level `@PreAuthorize` annotations provides defense in depth.

**Rationale**: **Defense in depth** security principle. If a future refactoring changes controller paths or adds new endpoints outside `/api/v1/admin/**`, the authorization checks could be accidentally bypassed. Method-level annotations make authorization explicit and independent of URL structure.

**Recommendation**: Consider adding `@PreAuthorize("hasRole('ADMIN')")` to admin controller classes as a secondary safety net. This is a **Nit** level suggestion as the current implementation is secure, but it improves resilience to future changes.

#### 6. **Rate Limiting Implementation** (5 submissions/hour)

Well-designed rate limiting with IP-based tracking:

```kotlin
private fun isRateLimitExceeded(ipAddress: String): Boolean {
    val oneHourAgo = Instant.now().minus(1, ChronoUnit.HOURS)
    val recentSubmissions = repository.countByIpAddressAndCreatedAtAfter(ipAddress, oneHourAgo)
    return recentSubmissions >= MAX_SUBMISSIONS_PER_HOUR
}
```

**Why this matters**: Rate limiting prevents abuse without requiring authentication. The sliding window approach is more fair than fixed windows. The constant `MAX_SUBMISSIONS_PER_HOUR` makes tuning easy.

#### 7. **IP Address Extraction** (`StoryController.kt:105-115`)

Proper handling of proxied requests with X-Forwarded-For fallback:

```kotlin
private fun extractIpAddress(request: HttpServletRequest): String? {
    val xForwardedFor = request.getHeader("X-Forwarded-For")
    if (!xForwardedFor.isNullOrBlank()) {
        return xForwardedFor.split(",").firstOrNull()?.trim()
    }
    return request.remoteAddr
}
```

**Why this matters**: Many production environments use reverse proxies (Cloud Run, GCP Load Balancer). Ignoring X-Forwarded-For would make rate limiting ineffective and enable trivial bypasses via proxy rotation.
