## Critical Issues

### 🔴 **CRITICAL**: Path Traversal Vulnerability in MDX File Writer

**File**: `apps/api/src/main/kotlin/com/nosilha/core/contentactions/services/MdxFileWriter.kt`
**Lines**: 92, 105-111

**Issue**: The `MdxFileWriter` constructs file paths using user-controlled slugs without validation, creating a critical path traversal vulnerability.

```kotlin
// Line 92
private fun resolveMdxPath(slug: String): Path = Path.of(contentPath, "$slug.mdx")

// Lines 106-111
private fun writeMdxFile(path: Path, content: String) {
    if (path.parent.notExists()) {
        Files.createDirectories(path.parent)  // Creates arbitrary directories!
    }
    Files.writeString(path, content, ...)
}
```

**Attack Vector**:
```kotlin
// Malicious slug: "../../../etc/passwd"
// Results in: Path.of("content/stories", "../../../etc/passwd.mdx")
// Writes to: /etc/passwd.mdx (or anywhere on filesystem)
```

**Fix**:
```kotlin
private fun resolveMdxPath(slug: String): Path {
    // Validate slug format
    if (!slug.matches(Regex("^[a-z0-9]+(?:-[a-z0-9]+)*$"))) {
        throw IllegalArgumentException("Invalid slug format: $slug")
    }

    // Construct path and verify it's within content directory
    val basePath = Path.of(contentPath).toAbsolutePath().normalize()
    val targetPath = basePath.resolve("$slug.mdx").normalize()

    // Prevent path traversal
    if (!targetPath.startsWith(basePath)) {
        throw SecurityException("Path traversal attempt detected: $slug")
    }

    return targetPath
}
```

**Impact**: Arbitrary file write vulnerability allowing attackers to:
- Write files anywhere on the filesystem
- Overwrite critical system files
- Execute code if combined with other vulnerabilities

**Principle**: Input validation + path normalization (OWASP A01:2021 - Broken Access Control)

### 🔴 **CRITICAL**: YAML Injection in MDX Frontmatter Parser

**File**: `apps/api/src/main/kotlin/com/nosilha/core/contentactions/api/AdminMdxController.kt`
**Lines**: 302-337

**Issue**: The `extractFrontmatter` function uses a naive regex-based YAML parser that's vulnerable to YAML injection attacks.

```kotlin
// Lines 315-329
yamlContent.lines().forEach { line ->
    val trimmed = line.trim()
    if (trimmed.isEmpty() || trimmed.startsWith("#")) return@forEach

    val parts = trimmed.split(":", limit = 2)
    if (parts.size == 2) {
        val key = parts[0].trim()
        val value = parts[1].trim().removeSurrounding("\"")  // Weak parsing!
        frontmatter[key] = value
        if (key == "slug") {
            slug = value  // User-controlled slug used in file path!
        }
    }
}
```

**Attack Vectors**:
1. **Slug injection**: `slug: "../../../etc/malicious"`
2. **Multi-line injection**: Values can contain newlines
3. **Type confusion**: All values stored as strings, no type validation

**Fix**:
```kotlin
import org.yaml.snakeyaml.Yaml
import org.yaml.snakeyaml.constructor.SafeConstructor

private fun extractFrontmatter(mdxSource: String): Pair<Map<String, Any>, String> {
    val frontmatterRegex = Regex("^---\\s*\\n(.*?)\\n---", RegexOption.DOT_MATCHES_ALL)
    val match = frontmatterRegex.find(mdxSource)
        ?: throw BusinessException("Invalid MDX format: frontmatter not found")

    val yamlContent = match.groupValues[1]

    // Use safe YAML parser
    val yaml = Yaml(SafeConstructor())
    val frontmatter = yaml.load<Map<String, Any>>(yamlContent)
        ?: throw BusinessException("Invalid YAML frontmatter")

    // Validate slug format
    val slug = frontmatter["slug"] as? String
        ?: throw BusinessException("Invalid MDX: slug field is required")

    if (!slug.matches(Regex("^[a-z0-9]+(?:-[a-z0-9]+)*$"))) {
        throw BusinessException("Invalid slug format: must be lowercase alphanumeric with hyphens")
    }

    return Pair(frontmatter, slug)
}
```

**Impact**:
- Path traversal via slug injection
- YAML injection attacks
- Data integrity issues from malformed YAML

**Principle**: Use established, safe parsers (OWASP A03:2021 - Injection)

### 🔴 **CRITICAL**: Missing Foreign Key Constraint Validation

**File**: `apps/api/src/main/resources/db/migration/V27__add_media_moderation.sql`
**Line**: 21

**Issue**: The `media_moderation_audit.performed_by` column references `users.id` but lacks a foreign key constraint, allowing orphaned records and data integrity violations.

```sql
-- Line 21
performed_by UUID NOT NULL,  -- No FK constraint!
```

**Impact**:
- Orphaned audit records when users are deleted
- Unable to identify who performed moderation actions
- Audit trail integrity compromised
- Violates database normalization principles

**Fix**:
```sql
performed_by UUID NOT NULL REFERENCES users(id) ON DELETE SET NULL,
```

**Alternative** (if audit immutability required):
```sql
performed_by UUID NOT NULL,
performed_by_email VARCHAR(255),  -- Denormalize email for immutable audit
FOREIGN KEY (performed_by) REFERENCES users(id) ON DELETE SET NULL,
```

**Principle**: Referential integrity (Database normalization, ACID properties)

### 🔴 **CRITICAL**: Race Condition in Slug Generation

**File**: `apps/api/src/main/kotlin/com/nosilha/core/contentactions/services/MdxGenerationService.kt`
**Lines**: 146-150

**Issue**: Slug generation doesn't check for uniqueness, creating potential race conditions and duplicate slugs.

```kotlin
private fun generateSlug(title: String): String =
    title
        .lowercase()
        .replace(Regex("[^a-z0-9]+"), "-")
        .trim('-')
```

**Attack Scenario**:
1. Admin A generates MDX for "My Story" → slug: `my-story`
2. Admin B simultaneously generates MDX for "My Story" → slug: `my-story`
3. Both commit to filesystem → file overwrite
4. Loss of data, archive corruption

**Fix**:
```kotlin
@Transactional
fun generate(story: StorySubmission, options: GenerateMdxRequest?): MdxContentDto {
    // Generate base slug
    var slug = generateSlug(story.title)

    // Check for existing archives with same slug
    var counter = 1
    var finalSlug = slug
    while (mdxArchiveRepository.findBySlug(finalSlug) != null) {
        finalSlug = "$slug-$counter"
        counter++
    }

    // Rest of generation logic using finalSlug
}

// Add to MdxArchiveRepository
fun findBySlug(slug: String): MdxArchive?
```

**Impact**:
- Data loss from file overwrites
- Archive corruption
- Unpredictable behavior in multi-admin scenarios

**Principle**: Concurrency control + uniqueness constraints (ACID properties)

### 🟠 **HIGH**: Missing Rate Limit Enforcement Mechanism

**File**: `apps/api/src/main/kotlin/com/nosilha/core/contentactions/services/DirectorySubmissionService.kt`
**Lines**: 72-79, 118-124

**Issue**: Rate limiting is implemented via database queries, which is inefficient and can be bypassed under high load.

```kotlin
// Lines 118-124
private fun isRateLimitExceeded(ipAddress: String): Boolean {
    val oneHourAgo = Instant.now().minusSeconds(3600)
    val recentSubmissions = directorySubmissionRepository
        .countByIpAddressAndCreatedAtAfter(ipAddress, oneHourAgo)
    return recentSubmissions >= MAX_SUBMISSIONS_PER_HOUR
}
```

**Problems**:
1. **Race condition**: Between check and save, multiple requests can pass
2. **DB load**: Every request triggers a COUNT query
3. **No distributed support**: Won't scale across multiple instances

**Fix**:
```kotlin
// Use Redis or in-memory rate limiter
@Autowired
private lateinit var rateLimiter: RateLimiter

@Transactional
fun submitDirectoryEntry(...): AdminDirectorySubmissionDto {
    if (ipAddress != null) {
        val key = "directory_submission:$ipAddress"
        if (!rateLimiter.tryAcquire(key, MAX_SUBMISSIONS_PER_HOUR, Duration.ofHours(1))) {
            throw RateLimitExceededException(...)
        }
    }
    // ...
}
```

**Recommended Libraries**:
- Bucket4j (token bucket algorithm)
- Spring Cloud Gateway rate limiter
- Redis-based distributed rate limiter

**Principle**: Defense in depth + scalability

### 🟠 **HIGH**: Unvalidated Redirect in Frontend Auth Flow

**File**: `apps/web/src/lib/backend-api.ts`
**Lines**: 104-109

**Issue**: The authentication error handler redirects to `/login` without validating the current URL, creating an open redirect vulnerability.

```typescript
if (response.status === 401) {
  await supabase.auth.signOut();
  window.location.href = "/login";  // Unvalidated redirect!
  throw new Error("Authentication expired. Please log in again.");
}
```

**Attack Vector**:
```
https://nosilha.com/directory?next=https://evil.com/phishing
→ Auth fails → Redirects to /login
→ User logs in → evil.com captures credentials
```

**Fix**:
```typescript
if (response.status === 401) {
  await supabase.auth.signOut();

  // Validate and sanitize redirect
  const currentPath = window.location.pathname;
  const allowedPaths = /^\/[a-zA-Z0-9\-\/]*$/;  // Only internal paths
  const returnUrl = allowedPaths.test(currentPath)
    ? encodeURIComponent(currentPath)
    : '';

  window.location.href = `/login${returnUrl ? `?returnUrl=${returnUrl}` : ''}`;
  throw new Error("Authentication expired. Please log in again.");
}
```

**Principle**: Input validation (OWASP A03:2021 - Injection)

### 🟡 **MEDIUM**: XSS Prevention Incomplete

**File**: `apps/api/src/main/kotlin/com/nosilha/core/shared/util/ContentSanitizer.kt`
**Lines**: 41-44

**Issue**: The `sanitizeStrict` method uses regex to strip HTML, which is prone to bypass.

```kotlin
}
```

**Bypass Examples**:

## Architecture & Code Quality Observations

### ✅ **POSITIVE**: Excellent Documentation
The code has exceptional inline documentation with Javadoc/KDoc comments explaining business rules, validation, and security considerations. This significantly improves maintainability.

### ✅ **POSITIVE**: Spring Modulith Event-Driven Architecture
The MDX file writing uses `@ApplicationModuleListener` for decoupling, which is a best practice for resilience (database commits succeed even if file I/O fails).

### ✅ **POSITIVE**: Content Sanitization
Input sanitization using OWASP HTML Sanitizer in `apps/api/src/main/kotlin/com/nosilha/core/shared/util/ContentSanitizer.kt` shows security awareness.

### ✅ **POSITIVE**: Just-in-Time (JIT) User Provisioning
The `UserSyncService` implements proper JIT provisioning with race condition handling.

The frontend has good error handling structure, but could benefit from more specific error types (e.g., `RateLimitError`, `ValidationError`) instead of generic `Error`.

### ⚠️ **IMPROVEMENT**: Transaction Boundaries
Some service methods lack `@Transactional` annotations where atomic operations are expected (verify in `MdxGenerationService` and related services).
